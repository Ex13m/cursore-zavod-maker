# /spec-interview — рождение SPEC.md через интервью

Режим: агент — интервьюер, директор — источник истины. Цель — SPEC.md,
по которому /work сможет строить без додумывания.

## Правила интервью

1. **По ОДНОМУ вопросу за раз.** Задал — молчи и жди ответа. Пакетные
   опросники запрещены.
2. **Порядок тем:** цель → пользователь → данные → границы → критерий успеха.
   Внутри темы можно 1–3 уточнения, потом дальше.
3. **Дави в слепые зоны.** Если ответ обходит риск (масштаб, деньги, стык
   с другой системой, приватность К0-данных) — задай неудобный вопрос прямо.
4. **Предлагай дефолты.** К каждому вопросу — разумный вариант по умолчанию.
   Ответ «сам реши» = принять дефолт и зафиксировать его в SPEC как решение.
5. **«Не знаю» — не дыра, а факт.** Заносится в секцию ОТКРЫТЫЕ. Ничего
   не выдумывать за директора.
6. **Потолок — 10 вопросов.** Дальше собирай SPEC из того, что есть.

## Структура SPEC.md

```markdown
# SPEC — <имя проекта>
## Цель            # что должно существовать и зачем (1 абзац)
## Пользователь    # кто пользуется, сценарий использования
## Данные          # откуда берутся, где живут, что приватно
## Границы         # что НЕ делаем в этой версии
## Критерий успеха # бинарный, проверяемый
## Решения         # принятые дефолты и выборы (с датой)
## ОТКРЫТЫЕ        # вопросы без ответа — вернуться до build
```

## Завершение

1. Показать собранный SPEC.md директору целиком.
2. С его ОК — закоммитить SPEC.md в репо проекта.
3. Предложить начать /work (фаза spec почти готова — берёт SPEC за основу).

## Когда ритуал не нужен

Мелочь/эксперимент на одну сессию — быстрый путь newproject без --spec
(пустой SPEC-шаблон заполняется по ходу, интервью не проводится).
