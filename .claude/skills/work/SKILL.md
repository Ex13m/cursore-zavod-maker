# /work — spec → build → review → distill

Дисциплина для ВСЕХ проектов KAIROS. Фазы идут строго по порядку,
следующая не начинается без завершения предыдущей.

## Фаза 1 — spec (план)

Написать и показать директору план из пяти блоков:

1. **Цель** — что должно существовать в конце, одним абзацем.
2. **Контекст** — что уже есть, на что опираемся, что нельзя ломать.
3. **Констрейнты** — рамки: технологии, лимиты, запреты, бюджет времени.
4. **КРИТЕРИЙ УСПЕХА** — измеримый и бинарный: «тесты проходят»,
   «3 виджета на экране», «файл собирается одной командой».
   Слабый критерий («сделай хорошо», «чтобы работало нормально») ЗАПРЕЩЁН —
   переформулировать до бинарного или дробить задачу.
5. **Формат вывода** — какие файлы/артефакты появятся и где.

Spec утверждается директором (ОК) до начала build.

## Фаза 2 — build (стройка)

- Строго следовать spec. Ничего не додумывать сверх него.
- Захотелось отступить от spec → остановиться, предложить правку spec,
  получить ОК, только потом продолжать.
- Работы вести шагами режима директора (объяснение → ОК → выполнение).

## Фаза 3 — review (проверка)

- Сверить результат с КРИТЕРИЕМ УСПЕХА из spec — буквально, пункт за пунктом.
- Не «кажется ок», а проверка: запустить, открыть, прогнать, показать вывод.
- Критерий не выполнен → вернуться в build с конкретным списком правок.
  Цикл build↔review крутится, пока критерий не выполнен.

## Фаза 4 — distill (выжимка, закрытие сессии)

- Предложить директору 1–3 урока/решения сессии, каждый одной строкой.
- С его ОК записать в `knowledge/lessons/` файлом: дата + контекст + урок.
- Без writeback сессия НЕ закрыта (правило KAIROS). Чат = черновик,
  наработка = файл в git.

## Чем /work НЕ является

- Это ручная дисциплина, НЕ автоматический overnight-loop (eval.json +
  pass rate) — тот живёт отдельным треком и сюда не подмешивается.
